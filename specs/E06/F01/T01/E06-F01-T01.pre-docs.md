## 2. Architecture Decisions

### 2.1 Base Class Choice: QMainWindow vs QWidget

**Decision**: Use `QMainWindow` as base class

**Rationale**:
- **Built-in Features**: `QMainWindow` provides menu bar, toolbar, status bar, and dock widget support
- **EDA Standard**: All professional EDA tools use main window pattern
- **State Persistence**: Built-in support for saving/restoring window geometry
- **Layout Management**: Automatic layout for central widget + docks + toolbars

**Alternatives Considered**:
- **QWidget**: Too basic, would need manual toolbar/menu/dock implementation
- **QDialog**: Not suitable for main application window
- **Custom window**: Unnecessary complexity, Qt's window management is robust

### 2.2 Window Sizing Strategy

**Decision**: Fixed default size (1600x900) with enforced minimum (1024x768)

**Rationale**:
- **1600x900 default**: Fits well on 1920x1080 displays (88% width, 83% height), leaving room for taskbar
- **Not maximized**: Users often want terminal/docs visible alongside
- **1024x768 minimum**: Common minimum for professional tools, ensures all UI elements readable
- **Future enhancement**: In E06-F05, will add adaptive sizing based on screen resolution

**Trade-offs**:
- Fixed size may not be optimal for 4K or small displays
- Mitigation: P1 enhancement will implement smart sizing
- For MVP, predictable sizing is more important than perfect adaptation

### 2.3 Window Flags Configuration

**Decision**: Explicitly set window flags for standard decorations

**Rationale**:
- **Explicit is better**: Some Linux window managers need explicit flags
- **Cross-platform**: Ensures consistent behavior across WMs
- **Debuggability**: Makes expected behavior clear in code

**Flags Set**:
```python
Qt.WindowType.Window                      # Top-level window
Qt.WindowType.WindowTitleHint             # Show title bar
Qt.WindowType.WindowSystemMenuHint        # Show system menu (right-click title)
Qt.WindowType.WindowMinimizeButtonHint    # Show minimize button
Qt.WindowType.WindowMaximizeButtonHint    # Show maximize button
Qt.WindowType.WindowCloseButtonHint       # Show close button
```

### 2.4 Initialization Pattern

**Decision**: Use private `_setup_window()` method for configuration

**Rationale**:
- **Separation of Concerns**: Keeps `__init__` clean and focused
- **Testability**: Can test setup logic independently
- **Extensibility**: Easy to add more setup methods (T02, T03, T04 will add more)
- **Standard Pattern**: Common in Qt applications

```python
from PySide6.QtWidgets import QMainWindow
from PySide6.QtCore import Qt, QSize


class InkMainWindow(QMainWindow):
    """Main application window shell for Ink schematic viewer."""

    def __init__(self):
        """Initialize the main window."""
        super().__init__()
        self._setup_window()

    def _setup_window(self):
        """Configure main window properties."""
        # Window title
        self.setWindowTitle("Ink - Incremental Schematic Viewer")

        # Default size (optimized for 1080p displays)
        self.resize(1600, 900)

        # Minimum size to prevent unusable layouts
        self.setMinimumSize(QSize(1024, 768))

        # Enable window decorations and standard controls
        self.setWindowFlags(
            Qt.WindowType.Window |
            Qt.WindowType.WindowTitleHint |
            Qt.WindowType.WindowSystemMenuHint |
            Qt.WindowType.WindowMinimizeButtonHint |
            Qt.WindowType.WindowMaximizeButtonHint |
            Qt.WindowType.WindowCloseButtonHint
        )
```

## 4. Technical Deep Dive

### 4.1 Window Sizing Calculations

#### 4.1.1 Default Size Rationale (1600x900)

**Target Display**: 1920x1080 (Full HD)

**Calculation**:
```
Width:  1600 / 1920 = 83.3% of screen width
Height:  900 / 1080 = 83.3% of screen height
```

**Reasoning**:
- Leaves ~160px on sides for other windows/desktop
- Leaves ~180px on top/bottom for title bar + taskbar
- User can still see terminal or documentation alongside
- Not aggressive (not maximized on first launch)
- Provides ample workspace for schematic viewing

**Alternative Sizes Considered**:
- **1920x1080 (maximized)**: Too aggressive, users may not want full screen
- **1400x800**: Too small, wastes space on modern displays
- **1280x720**: Old standard, too conservative for modern EDA work

#### 4.1.2 Minimum Size Rationale (1024x768)

**Calculation**:
```
Minimum width breakdown:
- Left hierarchy panel:   200px (minimum usable)
- Central canvas:         400px (minimum for schematic viewing)
- Right property panel:   200px (minimum usable)
- Splitters (2x):          10px
- Window chrome:          ~14px
Total:                   ~824px (rounded up to 1024px for safety)

Minimum height breakdown:
- Title bar:               ~30px
- Toolbar:                 ~40px
- Central area:           500px (minimum for usable schematic)
- Bottom message panel:   150px (minimum for log viewing)
- Status bar:              ~24px
- Splitters:               ~10px
Total:                   ~754px (rounded up to 768px for safety)
```

**Industry Standards**:
- Most EDA tools: 1024x768 or 1280x1024 minimum
- Qt Creator: 1024x768
- Visual Studio: 1024x768
- Eclipse: 1024x768

### 4.2 Window Flags Explained

```python
Qt.WindowType.Window
# - Creates top-level window (not child widget)
# - Has its own window frame
# - Can be minimized to taskbar

Qt.WindowType.WindowTitleHint
# - Shows title bar
# - Without this, no title or decorations

Qt.WindowType.WindowSystemMenuHint
# - Enables system menu (Alt+Space on Windows, right-click title on Linux)
# - Required for some window managers

Qt.WindowType.WindowMinimizeButtonHint
# - Shows minimize button in title bar
# - Allows user to minimize to taskbar

Qt.WindowType.WindowMaximizeButtonHint
# - Shows maximize button in title bar
# - Allows user to toggle maximized state

Qt.WindowType.WindowCloseButtonHint
# - Shows close button (X) in title bar
# - Allows user to close window
```

**Why Explicit Flags?**

Some window managers (especially on Linux) don't provide all decorations by default. Explicit flags ensure consistent behavior across:
- X11 window managers (i3, awesome, dwm)
- GNOME/KDE
- Wayland compositors

### 4.3 Qt Parent-Child Ownership

**Memory Management**:
- Qt uses parent-child ownership model
- Parent widget deletes all children when destroyed
- `InkMainWindow` has no parent (top-level), so must be explicitly deleted

**Lifecycle**:
```python
# Correct pattern
window = InkMainWindow()  # No parent
window.show()
# ... user closes window ...
# Qt automatically deletes window

# Or explicit cleanup
window = InkMainWindow()
window.show()
window.close()
window.deleteLater()  # Schedule for deletion
```

### 4.4 Platform-Specific Considerations

#### Linux (X11)
- Window decorations depend on window manager
- Some WMs need explicit flags
- Title bar height varies by theme
- No issues with standard Qt flags

#### Linux (Wayland)
- Client-side decorations (CSD) vs server-side (SSD)
- Qt handles automatically
- May see different title bar styles
- No code changes needed

#### Future (macOS/Windows)
- Different title bar heights
- Different window chrome sizes
- Qt handles cross-platform differences
- Current implementation is portable

## 5. Risk Analysis

### 5.1 Technical Risks

#### Risk 1: Window Doesn't Appear on Screen

**Impact**: Critical (blocker for all UI work)
**Probability**: Very Low
**Description**: Window creation succeeds but window not visible

**Root Causes**:
- QApplication not created before QMainWindow
- Window created but never shown
- Window positioned off-screen
- Display server issues (Wayland/X11)

**Mitigation**:
- Will be handled in T04 (proper QApplication creation)
- Add `window.show()` to test scripts
- Qt automatically positions windows on-screen by default
- Test on both X11 and Wayland

**Contingency**:
```python
# If positioning issues occur
def _setup_window(self):
    # ... existing setup ...

    # Center window on screen
    screen = QApplication.primaryScreen().geometry()
    self.move(
        (screen.width() - self.width()) // 2,
        (screen.height() - self.height()) // 2
    )
```

#### Risk 2: Minimum Size Not Enforced

**Impact**: Medium (poor UX but not broken)
**Probability**: Low
**Description**: User can resize window below minimum size

**Root Causes**:
- `setMinimumSize()` only enforced when window is visible
- Some window managers ignore size hints
- Qt bug or version incompatibility

**Mitigation**:
- Call `setMinimumSize()` before `show()`
- Test with window visible
- Test on multiple window managers

**Contingency**:
```python
def resizeEvent(self, event):
    """Enforce minimum size on resize."""
    super().resizeEvent(event)

    # Clamp to minimum if needed
    min_size = self.minimumSize()
    if self.width() < min_size.width() or self.height() < min_size.height():
        self.resize(
            max(self.width(), min_size.width()),
            max(self.height(), min_size.height())
        )
```

#### Risk 3: Window Decorations Missing

**Impact**: Low (cosmetic, window still functional)
**Probability**: Low
**Description**: Title bar or buttons not showing on some systems

**Root Causes**:
- Tiling window managers (i3, awesome) may not show decorations
- Wayland compositor differences
- User preference/theme

**Mitigation**:
- Accept that some WMs have different decoration styles
- Ensure close functionality still works (even without button)
- Document expected appearance

**Contingency**:
- User can still close with keyboard (Alt+F4) or WM commands
- Not a blocker for functionality
- Could add frameless mode in P2 for full control

### 5.2 Usability Risks

#### Risk 4: Default Size Not Suitable for All Displays

**Impact**: Low (user can resize)
**Probability**: Medium
**Description**: 1600x900 may be too large for small displays or too small for 4K

**Mitigation**:
- Minimum size ensures usability on small displays
- User can maximize on large displays
- P1 enhancement will add adaptive sizing
- Settings will remember user's preferred size

**Contingency**:
- Accept slight imperfection for MVP
- Priority 1 enhancement addresses this properly

#### Risk 5: Window Title Too Long

**Impact**: Very Low (cosmetic)
**Probability**: Very Low
**Description**: "Ink - Incremental Schematic Viewer" may be truncated in taskbar

**Mitigation**:
- Title is reasonable length (33 characters)
- Most taskbars show at least 20 characters
- Fallback: Qt truncates gracefully with ellipsis

**Contingency**:
- Could shorten to "Ink Schematic Viewer" (22 chars)
- Or just "Ink" in taskbar, full title in window
- Not critical for MVP
